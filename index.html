What is a bug?

Something we've all come across but what actually is a bug?
A software bug is an error, flaw, failure, or fault in a computer program or system that produces an incorrect or unexpected result, or causes it to behave in unintended ways.
http://en.wikipedia.org/wiki/Software_bug
This is what wikipedia describes it as. Quite concise and accurate. Contains a lots of words like failure, error and fault.

Why is it called a bug?

We all know the story about the moth right?

Before I started preparing for this talk I had this story in my head about a moth being caught in a relay in an early electromechanical computer. This did happen, here is a picture of the log book with the moth taped in. However, the phrase 'bug' was actually coined much earlier.

accually is Edison

The concept of errors in software dates back to Ada Lovelace notes on the analytical engine in 1843

Thomas Edison used the term 'bug' when referring to faults/difficulties in some of his inventions

It has been just so in all of my inventions. The first step is an intuition, and comes with a burst, then difficulties arise — this thing gives out and [it is] then that "Bugs" — as such little faults and difficulties are called — show themselves and months of intense watching, study and labor are requisite before commercial success or failure is certainly reached.
Thomas Edison 1878
Ada Lovelace (the worlds first computer scientist) first acknowledged the concept of errors in software when working on the analytical engine back in 1843. She noted that cards must be performed in the correct order or errors arise. However, it was Thomas Edison who first coined the term 'bug' when describing inexplicable defects in a letter to an associate in 1878.

Heisenbug

Heisenbug is a computer programming jargon term for a software bug that seems to disappear or alter its behavior when one attempts to study it.
http://en.wikipedia.org/wiki/Heisenbug
Named after the physicist Werner Heisenberg. He came up with the observer effect. This basically means that the act of observation will alter the state of what is being observed. For example checking the tyre pressure on a car, it is impossible to measure this without letting some air out of the tyre.

Often caused from debugging actually changing the program's operating environment

A Heisenbug is the kind of bug we don't want

Bohr bug

A repeatable bug; one that manifests reliably under a possibly unknown but well-defined set of conditions.
http://www.catb.org/jargon/html/B/Bohr-bug.html

The opposite of a Heisenbug

A Bohr bug is a bug that is easily repeatable

A Bohr bug is the kind of bug we want

Named after the bohr model of the atom. Renowned for its simplicity.

B.A.D. - Broken As Designed

Said of a program that is bogus because of bad design and misfeatures rather than because of bugginess.
http://www.catb.org/jargon/html/B/BAD.html
A program may have been implemented perfectly, exactly to spec, however if the spec was broken in the first place, then users may raise bugs.

Tree analogy

If a product, feature, architecture is poorly designed, you're gonna have a bad time

Schroedinbug

A design or implementation bug in a program that doesn't manifest until someone reading source or using the program in an unusual way notices that it never should have worked, at which point the program promptly stops working for everybody until fixed.
http://www.catb.org/jargon/html/S/schroedinbug.html
Schroedingers cat is a thought experiment devised by a physicist Erwin Schrodinger in 1935. It's the one where if a cat is in a box the cat is both dead and alive at some point until observed. So sometimes you may get a bug which doesn't manifest until someone looks further into it, realises it should never have worked and indeed stops working from then on in

Fractal bug

A bug, after which its resolution is found, reveals additional self-similar bugs elsewhere in the code, after which they are fixed, likewise appear elsewhere still.
http://c2.com/cgi/wiki?FractalBug
So you fix this bug, then the same bug pops up elsewhere, you fix that bug and another similar bug is reported, rinse and repeat.

Hindenbug

A bug with catastrophic behaviour

Not always a bug, can be from human error/mistake

Named after the Hindenburg disaster

Doesn't have to be a bug per se, any act that results in catastophic behaviour, dropping a live database for example. There is a scene in breaking bad where Walt and Jesse are out in the desert cooking meth and their van breaks down, thus leaving them stranded in the boiling hot desert, this could be explained as a hindenbug. Hindenbug is named after the Hindenburg disaster of 1937. This is where a german passenger airship caught fire and destroyed during it's attempt to dock.

Prevention > Cure


What can we do to prevent bugs?

Write less code

The only truly bug free code is the code which contains no code

Programming style

Defensive programming, coding standards, KISS, DRY etc.

Programming techniques

Can an exception be avoided? Is there a better alternative?

For example you may have a news feed or some 3rd party API integration in one portion of your site, if this API should go down for whatever reason, how is this handled? Would you really want to 500 the entire page for a small part? Probably not.

Take advantage of language features

Type hints, assertions, contracts etc.


Take advantage of IDE features

Linting, Inspections etc.

If your language supports these things, use them. They're there for a reason and can help stop bugs. Same with IDE, if you're IDE support inspections or similar, don't ignore them, they're trying to help.

Short, concise user stories

Rather than a monolithic specification, which is hard to understand, let alone implement.

Be agile, iterate

Code Analysis


Continuous integration

Automated testing

Mess detection, complexity analysis, copy paste analysis etc.

Anyone who's spent more than 5 minutes with Gav knows all about this. A *great* way of stopping bugs getting out the door.

Monitoring

Average response time, uptime, error logs etc.

Tools like New Relic etc.

You don't want to push software out there and forget about it, you want to monitor your application, keep on top of it and if you notice anything out of the ordinary, look at fixing before a bug report is received.

Communication


Ensure code comments are concise


Keep on top of documentation


Knowledge sharing


Shared ownership of code

You can be the smartest developer in the world, but if you keep all that knowledge in your head, you're no good to anyone and can in fact cause bugs. Other developers may no realise you were doing it *that* way, if you had mentioned it they wouldn't have done their thing *that other* way.

Education


Make things simple


Do everything you can to help the user

First and foremost, your application should be as simple to use as possible, but you should also support the user through things like user guides, support forums etc. where necessary. Helps cut down on ID10T bug reports.

Ditch IE (< 8)


Usage of IE7 has dipped below 1%


Get supported browsers locked down early

Not feasible for all project, but if you can get away with it, do it! There is little reason to be supporting IE8 and under if you can get away from it. Latest figures show just 0.8% of global internet use is done using IE6/7.

Managing bugs

Bugs are inevitable and should never be ignored

Bugs should be categorised and prioritised

Note, the severity of a bug is not the same as it's importance for fixing. Consider triage.

Use a good bug tracking tool (e.g. not bugzilla)

Use a good debugging tool

Ensure all bugs are managed in one place

How bugs are reported is key

Have a good bug report template or similar

Point 3 example: Windows BSOD may be considered severe, but if it happens very infrequently, a broken start icon may be considered more important. Point 6. Having separate support, ticket systems is fine, but bugs should be managed in one place.
